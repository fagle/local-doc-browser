# 产品定位与架构

## 1. 文档定位

本文描述可米 KomiOS 的产品定位、核心架构和长期演进方向。它不是单纯的文件预览器说明，而是用来统一后续功能取舍：我们要做的是面向 NAS / 私有存储的本地内容工作台，而不是另一个传统网盘或文件管理器。

## 2. 产品定位

可米 KomiOS 面向个人 NAS、Unraid、局域网共享盘和本机私有资料库。它的基础能力是浏览文档、源码、图片、音频和视频，但真正的目标是把私有存储变成一个可交互、可扩展、可持续使用的家庭/个人 webOS。

传统 NAS 和私有存储的弱点不在存储能力，而在操作界面：

- 文件夹树只能回答“东西在哪里”，不能回答“我现在要做什么”。
- 多数 NAS Web UI 偏管理后台，适合配置磁盘、用户和共享，不适合日常消费内容。
- 媒体、文档、代码、笔记、字幕、照片和归档资料通常被不同工具割裂。
- 用户在浏览器、播放器、文件管理器、下载器、笔记软件之间切换，私有存储本身没有形成一个有吸引力的主界面。

因此本项目的定位是：

> 在 NAS / 私有存储之上构建一个 webOS 式内容桌面，让文件不只是被存放，而是可以被打开、预览、播放、标记、恢复进度、搜索和组合成工作流。

## 3. 设计原则

### 3.1 私有优先

数据默认留在用户自己的机器、NAS 或局域网中。服务端直接读取本地挂载目录，不通过浏览器上传文件，也不依赖第三方云盘作为主路径。

### 3.2 Web 作为操作系统外壳

项目使用浏览器作为统一入口，但不把自己限制为“网页文件列表”。前端应该承担类似轻量 OS Shell 的角色：

- 提供工作区入口，而不是只提供目录输入框。
- 提供文件、媒体、文档和任务的统一预览区域。
- 记录最近打开、播放进度、索引状态和用户偏好。
- 逐步支持多窗口、侧边栏、小组件、任务队列、媒体库和插件式工具。

### 3.3 文件系统仍然是事实来源

文件仍然保存在真实目录中，SQLite 只做索引、缓存和状态记录。这样可以避免把用户锁进私有数据库，也方便用户继续用 SMB、NFS、SSH、Kodi、Jellyfin、Git、同步盘等工具访问同一批资料。

### 3.4 渐进增强

基础路径是“能打开、能预览、能搜索、能播放”。更高级的体验，如转码、缩略图、媒体墙、NFO 写回、AI 摘要、字幕处理、自动归类，都应该在基础文件访问之上渐进增强。

### 3.5 NAS 友好

NAS 上 CPU、GPU、磁盘和网络资源都可能有限。架构上要优先考虑：

- 扫描和探测结果可缓存。
- 视频转码可配置硬件加速或回退 CPU。
- UI 不因大目录、大媒体库和慢磁盘阻塞。
- 功能默认可用，但高消耗能力需要可关闭、可降级。

## 4. 当前架构

当前系统由三层组成：

```text
-----------------------------+
| Browser UI                 |
| index.html / styles.css    |
| app.js                     |
+-------------+---------------+
              |
              | HTTP API / static assets
              v
+-------------+---------------+
| Node Service                |
| dev-server.mjs              |
| auth / workspace / preview  |
| search / media probe        |
| transcode / NFO writeback   |
+-------------+---------------+
              |
              | fs / ffprobe / ffmpeg / sqlite
              v
+-------------+---------------+
| Private Storage             |
| local disk / NAS mount      |
| /workspace in Docker        |
| SQLite cache under /config  |
+-----------------------------+
```

### 4.1 Browser UI

前端负责提供日常操作界面：

- 工作区路径输入和最近打开。
- 文件/目录列表。
- 文档、源码、图片、PDF、音频、视频预览。
- HTML 预览/源码切换。
- 视频原始播放和转码播放切换。
- 主题、侧栏宽度、当前文件状态等本地交互状态。

它应该逐步从“左右分栏文件预览器”演进为“内容桌面”：文件列表只是入口，真正的核心是围绕内容产生的操作面板、媒体状态、搜索结果、任务队列和上下文工具。

### 4.2 Node Service

Node 服务是私有存储与 Web UI 之间的本地网关：

- 解析和记忆当前工作区。
- 提供登录和 session。
- 遍历目录、读取文件、返回预览内容。
- 维护 SQLite 索引缓存。
- 调用 `ffprobe` 探测媒体编码。
- 调用 FFmpeg 做转码播放。
- 将视频播放进度写回同名 Kodi `.nfo`。
- 在 Docker / Unraid 环境中映射 NAS 目录和持久化配置。

Node 层不应变成重型业务平台。它更像本地 OS 服务：对文件系统做安全包装，对高成本操作做缓存和队列化，对前端提供稳定 API。

### 4.3 SQLite Cache

SQLite 负责存储派生信息，而不是替代文件系统：

- 文件大小和修改时间缓存。
- 媒体编码探测结果。
- 后续可扩展缩略图索引、播放进度索引、搜索索引、标签、收藏、最近项目。

设计上要坚持“可重建”：如果 SQLite 丢失，用户文件不丢，系统可以重新扫描恢复大部分派生数据。

### 4.4 Docker / Unraid 部署

Docker 镜像把 Node 服务、FFmpeg 和运行环境打包起来。Unraid 部署脚本负责：

- 构建或运行容器。
- 挂载 NAS `/mnt/user` 到容器 `/workspace`。
- 持久化 `/config`。
- 配置端口、账号密码和转码加速。
- 写入 Unraid Docker 模板，便于后续在 UI 中管理。

这一层决定了项目不是只能在开发机跑，而是可以真正放到家庭 NAS 上作为日常入口。

## 5. webOS 式体验方向

这里的 webOS 不是指要实现完整操作系统，而是借用它的产品思想：用户打开浏览器后进入的是一个私有内容桌面，而不是一个冷冰冰的目录列表。

### 5.1 首页工作台

未来默认首页可以从“打开路径”演进为工作台：

- 最近继续看的视频。
- 最近打开的文档和目录。
- 常用工作区快捷入口。
- 媒体库、文档库、照片库、代码库入口。
- 后台任务状态，例如转码、扫描、生成缩略图。

### 5.2 内容应用化

不同类型的文件不只是预览，而是进入对应的轻应用：

- 视频：播放、转码、进度恢复、NFO、字幕、截图、媒体信息。
- 图片：相册、EXIF、Live Photo、旋转、缩略图墙。
- Markdown / 文档：阅读、复制源码、导出 HTML、全文搜索。
- 源码：语法高亮、目录浏览、README 优先、项目摘要。
- 压缩包：列表、预览、解压任务。

### 5.3 统一搜索与索引

搜索应该从“当前目录过滤”逐步演进到私有资料库搜索：

- 文件名搜索。
- 正文搜索。
- 媒体元数据搜索。
- 标签和收藏搜索。
- 按最近播放、最近修改、文件类型、工作区分组。

### 5.4 可玩性

NAS UI 常常弱，是因为它只强调管理，不强调探索。webOS 式设计可以加入更多让用户愿意打开的体验：

- 媒体墙、时间线、最近继续。
- 可拖拽/可停靠的侧栏和面板。
- 文件夹封面、目录摘要、智能推荐。
- “今日继续”“最近新增”“大文件”“未整理”等动态入口。
- 后续插件化工具，例如字幕处理、批量重命名、图片转换、资料归档。

可玩性不是花哨装饰，而是让私有存储从“仓库”变成“工作间”。

## 6. 和普通文件管理器的边界

本项目不追求替代系统文件管理器，也不应该把第一目标变成完整网盘：

- 不优先做复杂权限模型，单用户/家庭局域网是第一阶段重点。
- 不优先做在线协作、分享链和公网网盘能力。
- 不优先做全功能文件操作，如复杂复制、移动、冲突解决和回收站。
- 不把文件导入私有数据库，不破坏原始目录结构。

优先要做的是把“读、看、播、搜、继续、整理”做顺。

## 7. 演进路线

### 7.1 第一阶段：可靠的私有内容浏览器

- 完善文件预览能力。
- 稳定视频转码与原始播放切换。
- 完善 SQLite 缓存。
- 稳定 Docker / Unraid 部署。
- 处理大目录性能、认证、路径安全和异常提示。

### 7.2 第二阶段：媒体与文档工作台

- 首页工作台。
- 最近继续、收藏、标签。
- 缩略图和媒体墙。
- 全局搜索。
- 更完整的播放进度和媒体信息管理。

### 7.3 第三阶段：webOS Shell

- 多面板/多窗口体验。
- 应用化内容模块。
- 后台任务中心。
- 插件式工具入口。
- 面向 NAS 的资源监控和任务限流。

### 7.4 第四阶段：私有资料智能层

- 本地或可选远程 AI 摘要。
- 自动分类和去重建议。
- 视频字幕、章节、截图索引。
- 文档问答和语义搜索。
- 保持私有优先，所有智能能力都需要清晰标注数据流向。

## 8. 架构取舍

### 8.1 为什么使用 SQLite

SQLite 适合作为本地索引和缓存：

- 单文件部署，适合 Docker / NAS。
- SQL 查询方便，适合文件、媒体、标签、搜索索引等结构化数据。
- 工具链成熟，便于调试和迁移。
- 和文件系统配合自然，不需要引入独立数据库服务。

### 8.2 为什么保留真实文件路径

NAS / 私有存储的价值就在于用户拥有文件。真实路径带来几个好处：

- 可以继续被 SMB、NFS、Kodi、Jellyfin、Git、备份工具访问。
- 不需要导入/导出。
- 用户随时可以脱离本项目使用数据。
- 部署和恢复成本低。

### 8.3 为什么用 Web UI

Web UI 的优势是跨设备和 NAS 友好：

- PC、手机、平板、电视浏览器都能访问。
- Docker 部署后不需要安装客户端。
- 局域网分享入口简单。
- 前端可以逐步承载 webOS 式交互。

## 9. 成功标准

这个项目成功不只是“能列文件”，而是用户愿意把它作为私有存储的第一入口：

- 打开 NAS 时先打开它，而不是先打开 SMB 文件夹。
- 看视频能继续上次进度，必要时自动转码。
- 找资料能搜索，而不是手动翻目录。
- 文档、代码、图片、媒体都能在一个界面里被消费。
- 功能变多以后仍然保持私有、可控、可部署。

最终目标是让 NAS 从后台设备变成一个可进入、可探索、可持续扩展的个人 webOS。
